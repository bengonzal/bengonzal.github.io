ASCII Project

The purpose of this project is to create ascii art from a given image. Further explanation and examples can be seen down below, but the gist of this project is to take an image, apply a grayscale filter, map pixel values from the grayscale picture to an ascii charset, and recreate the image entirely out of ascii characters.

What problem does this solve?

I thought this idea was fun because I was able to experiment with pictures, and I liked running images through the program and zooming in to see the pictures made up of characters like #,@,_,+, etc. But if you want to talk about REAL WORLD problems...here are some answers ChatGPT helped me come up with:

  • Text Representation of Images - Images are typically stored as pixel data, and representing them in a text-based format (ASCII characters) can be useful for certain applications. This allows images to be displayed in environments or platforms that support only text, such as some command-line interfaces or text-based web browsers.
  • Aesthetic or Stylistic Rendering - Converting images to ASCII art can be seen as a way to provide a different, often stylized representation of an image. This can be used for artistic purposes or to create visually interesting effects, especially in contexts where traditional images might not be suitable.
  • Nostalgia and Cultural Appeal - ASCII art has a nostalgic appeal, reminiscent of early computer graphics and the ASCII art culture that emerged in the early days of computing. This can be leveraged to create content with a retro or vintage aesthetic.
  • Security and Privacy - In some cases, converting images to ASCII art could be used as a form of obfuscation, making it more challenging for automated systems to interpret or recognize the content of an image. This might have applications in security or privacy-sensitive scenarios.

Overview

As mentioned above, this project was created to transform an image to be re-created entirely out of ascii characters. Once the necessary environment requirements are fulfilled, a user can run the program and select an image to "ascii-tize" and end up with their very own image, entirely out of ascii characters. Given the resolution is kept and the proper media management software is used, the user can zoom in to view the individual ascii characters that compose the entire image and marvel at the little specs that come together to form something complete.

Key features

This program was written with mostly basic features in mind. I wanted the user to be able to customize how "detailed" they wanted their picture to be, so I added a good deal of user-options to fine tune the ascii-art generation until it was juuuuust right and ready to be saved. The following is what the user can find on the UI to help them dial everything in:

  1. Grayscale to ASCII Size Slider
    • This user-selected value can change the variable that maps the grayscale pixel values in the original image to the pixel values in the ascii charset.
    • Changing this value can have an interesting effect on how compact the ascii characters of the final image are.
  2. Scale Factor Slider
    • This user-selected value can change end-resolution of the "ascii-tized" image.
    • Changing this value can scale the original image UP or DOWN depending on the value, which results in a larger or smaller image to work with.
  3. Font Size Slider
    • This user-selected value can change the font-size of the ascii chars in the final image.
    • I've found that changing this value can result in "lighter" or "darker" images, depending on the size of the font you decide.

Examples further down show just how different images created with different values can be!

Design and Architecture

Designing/creating this project was fairly easy because there weren't too many "moving parts", so-to-say. The graphical user interface (GUI) was packaged in the same file as the backend implementation to make for an easy working file.

I did try my best to keep things modular while working on this project. I went back and fine-tuned functions/methods to keep things flowing nicely. If I were to have a bug in one method, it wouldn't affect the functionality of another method, which was the goal when keeping the program modular.

Overall, this program isn't too complicated, so not much needs to be talked about in the design and architecture aspect.

Technologies Used

This program was written entirely in Python. Any frameworks/libraries are ones that are easily found with a quick pip-install, so it should be accessible to use if you follow the directions in the README.

  • Tkinter
    • Tkinter is a built-in Python library used for creating graphical user interfaces (GUIs). The name "Tkinter" comes from "Tk Interface," as it is a Python interface to the Tk GUI toolkit. Tkinter provides a set of tools and widgets to help developers create desktop applications with graphical user interfaces.
  • PIL/Pillow
    • Pillow is a powerful Python Imaging Library (PIL) fork that serves as an open-source image processing and manipulation library. It provides extensive functionality for working with images, supporting various file formats and offering a wide range of operations, including opening, manipulating, and saving images.
  • Threading
    • Python provides a built-in threading module that allows you to create and manage threads. Each thread runs in its own separate space but shares the same data space, allowing threads to communicate with each other more easily than separate processes.

Future Plans

  1. Memory usage
    • This program can use quite a bit of memory, especially if you're using high-resolution images.
    • In the future, I want to try and make it memory efficient to generate ascii-art images so more users have accessibility to see my work!
  2. UI Changes
    • I'm trying to make changes to make the GUI more easily-usable. If your images are too big, most of the interface gets taken up by the original picture, and you don't see what the final image looks like until you already save it, so I'm working towards making the UI interactive where you can zoom in and out without leaving the application.
  3. Java?
    • Probably the biggest thing I'm working on with the future of this project is porting it over to the Java programming language.
    • Java is more easily-usable in terms of deployment (at least I think so), so I want it to be as easy as downloading the repo and clicking on an executable file, no commands or external environments required!
